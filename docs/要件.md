# 必要なカスタムフィールド

NetBox管理画面で、DCIM > Cable に対して以下のカスタムフィールドが作成済みである前提とする。

## 1. barcode

- name: `barcode`
- label: `バーコード`
- type: Text
- 対象モデル: DCIM > Cable
- 必須: 任意
- 一意性: 運用上は一意にすることを推奨する
- 保存値例: `CBL-000001`

## 2. cable_status

ステータス管理には、`is_configured` および `is_laid` の2フィールドは使用しない。  
代わりに、Selection型の単一カスタムフィールド `cable_status` を使用する。

- name: `cable_status`
- label: `ケーブルステータス`
- type: Selection
- 対象モデル: DCIM > Cable
- 必須: 任意
- default: `not_created`

選択肢は以下とする。

| value | label |
|---|---|
| `not_created` | `未作成` |
| `configured` | `作成済み` |
| `laid` | `敷設済み` |

ステータスは、更新権限を持つユーザーであれば以下のいずれの状態にも自由に変更可能とする。

- `not_created`
- `configured`
- `laid`

ステータス遷移の前進のみ制限は行わない。

---

# カスタムフィールド未作成時の挙動

必要なカスタムフィールド `barcode` または `cable_status` が存在しない場合、500エラーで落とさないこと。

設定不備を示すJSONエラーを返すこと。

例:

- HTTPステータス: `500`
- code: `custom_field_not_configured`
- error: `Required custom field 'cable_status' is not configured.`

画面上ではBootstrap Alertで設定不備が分かるように表示すること。

# バックエンド要件

## 1. プレフィックスルーティング設計

スキャンした値が `CBL-` から始まる場合は、ケーブル用の処理へルーティングする。

将来的にL3スイッチ等の機器バーコード、例: `DEV-`、を追加できるよう、プレフィックスごとの処理を容易に分岐追加できる設計にすること。

無効なプレフィックスの場合は、適切なJSONエラーを返すこと。

---

## 2. バーコード値の正規化ルール

スキャン値および手入力値は、必ず同じ正規化・バリデーション・検索処理を通すこと。

正規化ルールは以下とする。

- 前後の空白は `strip()` 相当で除去する
- 大文字小文字は区別しない
- 検索時はスキャン値全体を使用する
- `CBL-` の後ろだけを切り出して検索しない
- `label` はcase-insensitiveな完全一致とする
- `custom_field_data["barcode"]` もcase-insensitiveな完全一致とする
- 部分一致検索は行わない
- 前後空白除去後に空文字の場合は400エラーを返す
- 入力値の最大長は128文字とする
- 最大長を超えた場合は400エラーを返す

文字種の正規表現制限は設けない。  
ただし、プレフィックスによるルーティング判定は行うこと。

---

## 3. Cable検索仕様

対象のCableオブジェクトは、正規化後のスキャン値から取得する。

検索対象は以下とする。

- `Cable.label`
- `Cable.custom_field_data["barcode"]`

検索は完全一致とし、大文字小文字は区別しない。

`label` と `custom_field_data["barcode"]` の両方に一致した場合でも、同一Cableであれば1件として扱う。

異なるCableが2件以上一致した場合は、最初の1件を返さず、409 Conflict相当のJSONエラーを返すこと。

例:

- Cable Aの `label` が `CBL-0001`
- Cable Aの `barcode` も `CBL-0001`

この場合は同一Cableなので1件扱いとする。

一方で以下の場合は409 Conflictとする。

- Cable Aの `label` が `CBL-0001`
- Cable Bの `barcode` が `CBL-0001`

該当Cableが存在しない場合は404エラーを返すこと。

---

## 4. 配線トレース

NetBoxの `trace()` メソッドを使用し、中間のパッチパネル等の経路情報すべてと、最終的な末端の接続先、アクティブ機器のポートなどを取得してJSONレスポンスに含めること。

Cableのtrace結果は、A端側とB端側を分けて返すこと。

接続先の表示は、物理的接続先と論理的接続先を区別すること。

- **接続先サマリー** には、ケーブルが直接接続されている**物理的なポート**（このケーブル自身のA端/B端 termination）を表示すること（`endpoints`）。
- **経路トレース** には、`trace()` でパッチパネル等を貫通して到達する**論理的な最終接続先**までの経路を表示すること（`trace` / `trace_endpoints`）。

trace結果のJSON表現は以下のような構造とする。

- `trace.a_side`
- `trace.b_side`

各要素は以下の共通形式に正規化すること。

| field | 内容 |
|---|---|
| `name` | 表示名 |
| `type` | `interface` / `frontport` / `rearport` / `cable` 等の種別 |
| `device` | デバイス名。該当しない場合は `null` |
| `url` | NetBox上の該当オブジェクトURL。取得できない場合は `null` |

未接続の端は空配列とする。  
未接続はエラー扱いしない。

可能な場合、対象Cableおよびtrace内の各オブジェクトにはNetBox標準の詳細画面URLを含めること。

# 権限要件

## 1. 認証

プラグイン画面およびAPIは、NetBoxの標準セッション認証を使用する。

未ログインユーザーは利用できない。

---

## 2. 閲覧権限

プラグインのスキャン画面自体は、ログイン済みユーザーであればアクセス可能とする。

Cable照会時はNetBox標準のObject Permissionを尊重し、対象Cableに対する `dcim.view_cable` 権限を持つユーザーのみ詳細情報を取得できるようにする。

運用上は、全ログインユーザーが閲覧できるように、全一般ユーザー相当のグループまたは対象ユーザーに `dcim.view_cable` のObject Permissionを付与する。

これにより、実運用上はログインユーザー全員が閲覧可能になるようにする。

---

## 3. 更新権限

更新権限の判定には、NetBox標準のObject Permissionを使用する。

更新可否は以下とする。

- スーパーユーザーは常に更新可能
- 一般ユーザーは、対象Cableに対して `dcim.change_cable` 権限を持つ場合のみ更新可能

バックエンドは、照会レスポンスに `can_update` フラグを含めること。

- 更新可能な場合: `can_update: true`
- 更新不可の場合: `can_update: false`

フロントエンドは、`can_update` が `true` の場合のみステータス更新用のドロップダウンと更新ボタンを表示する。

`can_update` が `false` の場合は、ステータスを閲覧表示のみとする。

---

# ステータス更新処理

ステータス更新では、Cableのカスタムフィールド `cable_status` の値のみを更新する。

更新可能な値は以下のみとする。

- `not_created`
- `configured`
- `laid`

不正な値が指定された場合は400エラーを返すこと。

更新時は、NetBox標準の `change_logging` コンテキストマネージャを使用し、Changelogにユーザーの操作ログを必ず記録すること。

保存前に `custom_field_data["cable_status"]` へ新しい値を設定し、NetBox標準のバリデーションを通すこと。

同時更新時の排他制御は行わず、標準仕様として後勝ちとする。

更新成功時は、更新後の `cable_status` のvalueおよび表示ラベルをJSONで返すこと。

# API要件

APIエンドポイントの詳細URL、HTTPメソッド、リクエストJSON、成功レスポンスJSONの完全な仕様は別途定義する。

ただし、以下の共通要件を満たすこと。

- API URLは `/plugins/barcode/api/` 配下に配置すること
- Fetch APIから利用すること
- JSONリクエストを受け取ること
- JSONレスポンスを返すこと
- CSRF保護を有効にすること
- 更新系リクエストではDjangoのCSRFトークンをCookieから取得し、リクエストヘッダーに含めること
- ログイン済みセッションを前提とすること

# エラーハンドリング要件

エラー時はJSON形式で返すこと。

エラーJSONの形式は以下に統一する。

| field | 内容 |
|---|---|
| `success` | 常に `false` |
| `error` | 人間向けメッセージ |
| `code` | 機械処理向けエラーコード |

例:

{
  "success": false,
  "error": "Cable not found.",
  "code": "cable_not_found"
}

使用するHTTPステータスコードは以下とする。

| HTTPステータス | 用途 |
|---|---|
| 400 | 不正なJSON、不正なプレフィックス、不正な値、空文字、最大長超過 |
| 403 | 権限なし |
| 404 | 該当Cableなし |
| 409 | 複数Cable一致 |
| 405 | 許可されていないHTTPメソッド |
| 500 | 想定外エラー、必須カスタムフィールド未設定などのサーバー側設定不備 |

代表的なエラーコードは以下とする。

- `invalid_json`
- `invalid_prefix`
- `invalid_barcode`
- `barcode_too_long`
- `cable_not_found`
- `multiple_cables_found`
- `permission_denied`
- `invalid_status`
- `custom_field_not_configured`
- `method_not_allowed`
- `internal_error`

画面上では、エラー内容をBootstrap Alertで表示すること。

# フロントエンド要件

## 1. バーコードスキャン画面

NetBoxのベーステンプレート `base/layout.html` を継承し、レスポンシブなUIを構築すること。

`html5-qrcode` を使用し、スマートフォンの背面カメラ、`environment`、でCode128のバーコードを読み取ること。

カメラ利用にはHTTPSまたはlocalhostが必要である。  
カメラ起動に失敗した場合は、手入力フォームを表示すること。

手入力フォームで入力された値も、カメラで読み取った値と同じ正規化・バリデーション・検索処理を通すこと。

---

## 2. html5-qrcodeの配置

`html5-qrcode` はCDN経由で読み込まないこと。

プラグインのstatic配下にローカル配置すること。

配置例:

- `netbox_barcode_plugin/static/netbox_barcode_plugin/js/html5-qrcode.min.js`

第三者ライブラリのライセンス表記は、`THIRD_PARTY_LICENSES.md` または `NOTICE` に含めること。

---

## 3. トーチ制御

暗所作業を想定し、カメラストリームのトーチ、フラッシュライト、をON/OFFするボタンを配置すること。

トーチ機能が利用できないブラウザ・端末では、トーチボタンをdisabledにし、以下のメッセージを表示すること。

- `この端末ではライト制御に対応していません`

トーチ機能の失敗は、スキャン全体の失敗扱いにしないこと。

---

## 4. スキャン結果表示と操作UI

スキャン成功時は即座にカメラを停止し、Fetch APIでバックエンドへデータを照会すること。

自動連続スキャンは行わない。

取得した情報を視認性の高いUIで表示すること。

表示対象は以下とする。

- ケーブル名
- ケーブルのNetBox詳細画面へのリンク
- 現在のケーブルステータス
- 接続先サマリー（物理的な直結ポート）
- A端側の経路情報
- B端側の経路情報
- 中間経路のリスト
- 末端の接続先（論理的な最終接続先）
- trace内の各オブジェクトのNetBox詳細画面URL、取得可能な場合

バックエンドから渡された `can_update` が `true` の場合のみ、ステータス更新用のドロップダウンと更新ボタンを表示すること。

`can_update` が `false` の場合は、ステータスをテキストとして閲覧表示すること。

該当データなし、権限なし、通信エラー、設定不備などの場合は、Bootstrapのアラートコンポーネントを使用して画面上に通知を表示すること。

画面下部の「再スキャン」ボタンがタップされた場合にのみ、カメラ待機状態を再開させること。

---

## 5. ステータス更新UI

ステータス更新用ドロップダウンには以下の選択肢を表示すること。

| 表示名 | value |
|---|---|
| 未作成 | `not_created` |
| 作成済み | `configured` |
| 敷設済み | `laid` |

更新ボタン押下時、Fetch APIを使用してバックエンドに更新リクエストを送信すること。

DjangoのCSRFトークンをCookieから取得して、リクエストヘッダーに含めること。

更新成功・失敗の結果は、画面の再読み込みを行わずにBootstrap Alertとして画面上部に動的生成して表示すること。
